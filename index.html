Case Study · Blog CMS System

Designing a backend system using a modular monolith approach

Context

This project is a backend-focused blog system I am building to develop a deeper understanding of how real-world applications are structured beyond basic CRUD implementations. The goal was to move from frontend-focused development toward system-level thinking and backend architecture.

Problem

Many beginner projects focus on implementing features without a clear system structure, which leads to tightly coupled code and poor scalability. I wanted to understand how to design an application where responsibilities are clearly separated and the system can evolve without becoming difficult to maintain.

Approach

I designed the system as a modular monolith, organizing it into clear layers: API (routes), service layer (business logic), and repository layer (data access). Each request follows a structured flow, ensuring that responsibilities are isolated and the system remains testable.

Instead of starting with authentication, I began with the content domain (posts) to establish a solid foundation for data flow and business logic before introducing cross-cutting concerns like security.

Key Decisions

I chose a modular monolith architecture to balance simplicity and scalability. This avoids the complexity of microservices while still allowing the system to be structured into independent domains such as content, authentication, and engagement.

The service and repository pattern was used to separate business logic from data access, improving maintainability and making the system easier to extend as new features are added.

Current State

The system currently supports post CRUD operations with request validation, structured API responses, and error handling. I am working on completing the data persistence layer and aligning schemas with database models.

What I Learned

This project shifted my approach from building features to designing systems. I gained a clearer understanding of how layering, domain separation, and incremental complexity contribute to maintainable backend applications.

Case Study · Career Recommendation Web App

Contributing to a data-driven recommendation system in a team environment

Context

This project was developed as a final-year capstone, where our team built a web application to guide users toward suitable career paths based on behavioral assessments, structured data, and recommendation logic.

Problem

Students often struggle to choose career paths due to limited access to structured information and a lack of personalized guidance. The goal was to design a system that could provide meaningful recommendations based on user behavior, personality, and educational data.

My Role

I focused on the data and logic layer of the system, contributing to how the application generates relevant recommendations.

This included collecting and structuring datasets of universities and courses, designing a user survey to understand decision-making patterns, and supporting the development of personality-based recommendation logic using a DISC model.

Approach

I worked on ensuring that the data used by the system was accurate and structured in a way that could support meaningful recommendations. The survey design helped capture user needs and influenced how features were shaped within the application.

For the recommendation logic, I contributed to aligning personality traits with career paths, helping define how user inputs translate into system outputs.

Outcome

The result was a system capable of generating personalized career recommendations, supported by structured data and user-driven inputs. My contributions helped improve the relevance and usability of these recommendations.

What I Learned

This project gave me experience working within a team on a larger system, where different components depend on each other. I learned how data quality, user input, and logic design directly impact the effectiveness of application features.

It also strengthened my ability to contribute to a system even when not working across every layer, focusing instead on making specific parts of the system more effective.
